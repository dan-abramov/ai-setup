# Фича 002: Генерация краткого описания задачи перед переходом к решению

## Ссылки
- Brief: `./brief.md`
- https://github.com/dan-abramov/ai-setup/issues/5

## Проблема и цель
Сейчас после ввода `skill/topic` к решению задачи переходит `0%` пользователей. 
Цель фичи: к **1 мая 2026** обеспечить переход к решению не менее чем у `80%` пользователей.

## Scope
Входит:
- Генерация текстового описания задачи по `skill` и `topic` для валидного запроса.
- Валидация сгенерированного текста по длине, наличию `skill/topic` и отсутствию HTML.
- Переход пользователя к шагу решения после успешной генерации.
- Отображение состояний `EMPTY`, `LOADING`, `SUCCESS`, `ERROR`.

Не входит:
- Изменение логики проверки пользовательского решения.
- Изменение генерации автотестов для пользовательского решения.
- Новые рекомендации навыков/тем, персонализация, история запросов.

## Границы по модулям (<=3)
1. `Generation::BuildDescriptionService` (генерация текста).
2. `Generation::DescriptionValidator` (проверка инвариантов результата).
3. `GenerationFlow` (контроллер/endpoint + UI шаг перехода к решению).

## Термины
- `valid request`:
  - `skill` и `topic` после `strip_tags + trim` имеют длину `1..100`.
- `valid description`:
  - после `strip_tags + trim` длина `1..150` символов;
  - содержит подстроку `topic` (case-insensitive);
  - содержит подстроку `skill` (case-insensitive);
  - содержит явный призыв: фрагмент `Реши через`.

## Функциональные требования
- `FR-1`: при валидном запросе система возвращает `task_description` и разрешает переход к решению.
- `FR-2`: `task_description` должен соответствовать `valid description`.
- `FR-3`: при невалидном запросе или невалидном результате генерации переход к решению блокируется.
- `FR-4`: при ошибке генерации пользователь видит код ошибки и действие `Повторить`.

## Нефункциональные требования
- `NFR-1` (latency): для валидных запросов `p95 <= 1s` от submit до ответа `SUCCESS`, окно измерения — последние 200 валидных запросов.
- `NFR-2` (success rate): не менее `95%` валидных запросов завершаются `SUCCESS`.
- `NFR-3` (business KPI): доля пользователей, перешедших к решению после ввода `skill/topic`, не ниже `80%` к **1 мая 2026**.

## Состояния (Uniform)

| State | Условие входа | UI/поведение | Условие выхода |
|---|---|---|---|
| `EMPTY` | Шаг открыт, генерация ещё не запускалась | Кнопка перехода к решению недоступна, результата нет | Пользователь запускает генерацию |
| `LOADING` | Отправлен валидный submit | Показан индикатор загрузки, повторный submit заблокирован | Ответ сервиса -> `SUCCESS` или `ERROR` |
| `SUCCESS` | Получен `valid description` | Показан текст задачи и активна кнопка `Перейти к решению` | Пользователь переходит к решению |
| `ERROR` | Любая ошибка из таблицы ниже | Показан код/сообщение ошибки; для `E204-E209` доступна кнопка `Повторить`, для `E201-E203` кнопка `Повторить` недоступна до исправления входа | Повторный submit или исправление входных данных |

## Сценарии ошибок

| Код | Условие | Сообщение | Retry |
|---|---|---|---|
| `E201` | `skill` пустой после нормализации | `Укажите навык` | Нет |
| `E202` | `topic` пустой после нормализации | `Укажите тему` | Нет |
| `E203` | `skill` или `topic` длиннее 100 | `Некорректная длина skill/topic` | Нет |
| `E204` | Таймаут генерации | `Не удалось сгенерировать описание вовремя` | Да |
| `E205` | Ошибка провайдера/внутренняя ошибка | `Ошибка генерации описания` | Да |
| `E206` | Результат пустой после `strip_tags + trim` | `Описание задачи пустое` | Да |
| `E207` | Длина результата > 150 | `Описание задачи превышает 150 символов` | Да |
| `E208` | В результате нет `topic` | `Описание не содержит тему` | Да |
| `E209` | В результате нет `skill` или нет фразы `Реши через` | `Описание не содержит просьбу решить через навык` | Да |

## Acceptance Criteria (testable)
- `AC-01`: при валидном `skill/topic` endpoint возвращает `200`, состояние `SUCCESS`, и `task_description`.
- `AC-02`: `task_description` после `strip_tags + trim` имеет длину `<=150`.
- `AC-03`: `task_description` содержит `topic` (case-insensitive).
- `AC-04`: `task_description` содержит `skill` (case-insensitive).
- `AC-05`: `task_description` содержит фразу `Реши через`.
- `AC-06`: при `E204-E209` возвращается `422`, переход к решению недоступен, UI в `ERROR`, доступен `Повторить`.
- `AC-07`: при `E201-E203` возвращается `422`, переход к решению недоступен, UI в `ERROR`, `Повторить` недоступен до исправления входа.
- `AC-08`: для набора из 200 валидных запросов `p95 <= 1s` и success rate `>=95%`.

## Инварианты
- Переход к решению возможен только из состояния `SUCCESS`.
- На шаг решения передаётся ровно один `task_description` для одного submit.
- `task_description` всегда plain text без HTML после нормализации.
- Длина `task_description` после нормализации всегда `<=150`.
- Текущая фича не меняет проверку пользовательского решения и не меняет генерацию тестов решения.

## Ограничения реализации
- Не добавлять новые gem-зависимости без согласования.
- Использовать текущий Rails стек и паттерн service objects.
- Не выходить за границы 3 модулей, перечисленных в разделе «Границы по модулям».
- Не изменять файлы вне проекта `ai-setup`.
- Генерация текста-описания выполняется через `Generation::AiClient`.
  Провайдер: `OpenRouter` (бесплатные модели).
  Primary model: `qwen/qwen2.5-7b-instruct:free`.
  Fallback model: `meta-llama/llama-3.1-8b-instruct:free`.
  Если primary и fallback недоступны/вернули ошибку — код `E205`.
  HTTP-клиент: стандартный `Net::HTTP` (без новых gem).

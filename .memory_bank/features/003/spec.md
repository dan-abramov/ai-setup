# Фича 003: Повторное открытие последней сгенерированной задачи после закрытия вкладки

## Ссылки
- Brief: `./brief.md`
- https://github.com/dan-abramov/ai-setup/issues/7

## Проблема и цель
Сейчас после закрытия вкладки пользователь теряет доступ к уже сгенерированному описанию задачи.
Цель фичи: к **1 мая 2026** обеспечить возможность повторного открытия последней успешной задачи не менее чем в `95%` сессий.

## Scope
Входит:
- Создание новой модели `Task` с полем `description`.
- Запись результата успешной генерации в `tasks.description`.
- Переход пользователя после успешной генерации на страницу `GET /task/:id`.
- Возможность повторно открыть ту же задачу по адресу `GET /task/:id` после закрытия вкладки.
- Обработка ошибок при создании/открытии `Task` с кодами ошибок.

Не входит:
- Изменение логики генерации описания задачи и кодов `E201-E209`.
- Изменение проверки пользовательского решения и генерации автотестов.
- История из нескольких задач, список «все прошлые задачи», поиск и фильтры.

## Границы по модулям (<=3)
1. `GenerationRequests` (контроллер + service object) — создание `Task` после успешной генерации и выдача перехода на `GET /task/:id`.
2. `Task` (model + controller + show view) — хранение и повторное открытие задачи.
3. `routes.rb` — маршруты `POST /generation_requests` и `GET /task/:id`.

## Термины
- `valid request`:
  - `skill` и `topic` после `strip_tags + trim` имеют длину `1..100`.
- `valid description`:
  - результат генерации проходит текущие проверки `E201-E209` (без изменений логики).
- `reopenable task`:
  - запись `Task` существует;
  - `description` не пустой.

## Функциональные требования
- `FR-1`: после успешной генерации система создаёт `Task`.
- `FR-2`: в `Task.description` сохраняется сгенерированное описание задачи.
- `FR-3`: после создания `Task` пользователь переходит на `GET /task/:id`.
- `FR-4`: при ошибках валидации входа (`E201-E203`) `Task` не создаётся.
- `FR-5`: при ошибках генерации/валидации описания (`E204-E209`) `Task` не создаётся.
- `FR-6`: `GET /task/:id` открывает ранее созданную задачу без повторной генерации.
- `FR-7`: если `Task` не найден или невалиден для открытия, пользователь получает предсказуемую ошибку.

## Нефункциональные требования
- `NFR-1` (business KPI): не менее `95%` сессий должны позволять повторно открыть уже созданную задачу к **1 мая 2026**.
- `NFR-2` (latency): `p95 <= 1s` для шага `submit -> redirect to /task/:id` на валидных запросах (окно 200 последних валидных запросов).
- `NFR-3` (backward compatibility): текущая логика кодов `E201-E209` остаётся неизменной.

## Состояния (Uniform)

| State | Условие входа | UI/поведение | Условие выхода |
|---|---|---|---|
| `EMPTY` | Форма открыта, submit не выполнялся | Отображаются поля `skill/topic`, ошибки отсутствуют | Пользователь нажимает `Сгенерировать` |
| `LOADING` | Начат submit валидного запроса | Повторный submit заблокирован | Ответ сервера -> `SUCCESS` или `ERROR` |
| `SUCCESS` | Успешно создан `Task` | Выполнен переход на `GET /task/:id`, описание отображено | Пользователь закрывает вкладку или переходит дальше |
| `ERROR` | Любая ошибка из таблицы ниже | Показан код ошибки и сообщение; для retryable ошибок доступно `Повторить` | Повторный submit или исправление входа |

## Сценарии ошибок

| Код | Условие | Сообщение | Retry | Создание Task |
|---|---|---|---|---|
| `E201` | `skill` пустой после нормализации | `Укажите навык` | Нет | Нет |
| `E202` | `topic` пустой после нормализации | `Укажите тему` | Нет | Нет |
| `E203` | `skill` или `topic` длиннее 100 | `Некорректная длина skill/topic` | Нет | Нет |
| `E204` | Таймаут генерации | `Не удалось сгенерировать описание вовремя` | Да | Нет |
| `E205` | Ошибка провайдера/внутренняя ошибка | `Ошибка генерации описания` | Да | Нет |
| `E206` | Результат пустой после нормализации | `Описание задачи пустое` | Да | Нет |
| `E207` | Длина результата > 150 | `Описание задачи превышает 150 символов` | Да | Нет |
| `E208` | В результате нет `topic` | `Описание не содержит тему` | Да | Нет |
| `E209` | В результате нет `skill` или фразы `Реши через` | `Описание не содержит просьбу решить через навык` | Да | Нет |
| `E301` | Не удалось сохранить `Task` в БД | `Не удалось создать задачу` | Да | Нет |
| `E302` | `GET /task/:id` для несуществующего id | `Задача не найдена` | Нет | Не применимо |
| `E303` | `Task` найден, но `description.blank?` | `Задача недоступна для открытия` | Нет | Не применимо |

## Acceptance Criteria (testable)
- `AC-01`: при валидном `skill/topic` и валидном результате генерации создаётся одна запись `Task` и выполняется переход на `GET /task/:id`.
- `AC-02`: `tasks.description` после нормализации имеет длину `<=150` и соответствует текущим правилам валидного описания.
- `AC-03`: при `E201-E209` или `E301` ответ содержит код ошибки, состояние `ERROR`, и `Task.count` не увеличивается.
- `AC-04`: `GET /task/:id` существующей задачи возвращает `200` и отображает сохранённый `description`.
- `AC-05`: `GET /task/:id` для несуществующего id возвращает ошибку `E302`.
- `AC-06`: `GET /task/:id` для задачи с пустым `description` возвращает ошибку `E303`.
- `AC-07`: после закрытия вкладки пользователь может повторно открыть ту же задачу по её URL `GET /task/:id` без повторной генерации.
- `AC-08`: на окне последних 200 сессий, где уже была создана `Task`, доля успешного повторного открытия `GET /task/:id` не ниже `95%`.

## Инварианты
- `Task` создаётся только после успешной генерации и прохождения проверок описания.
- Для одного успешного submit создаётся ровно один `Task`.
- При любой ошибке до `SUCCESS` запись `Task` не создаётся.
- Повторное открытие задачи не инициирует новую генерацию.
- Контракт ошибок `E201-E209` не изменяется этой фичей.

## Ограничения реализации
- Не добавлять новые gem-зависимости без согласования.
- Использовать MVC и service objects.
- Затрагивать только 3 модуля из раздела «Границы по модулям».
- Не изменять файлы вне проекта `ai-setup`.
- Не изменять существующий контракт и бизнес-логику генерации описания (`E201-E209`).

## Проверка критериев качества
- `Testable`: критерии `AC-01..AC-08` проверяются request/system/model тестами.
- `Ambiguous-free`: отсутствуют слова вида «быстро», «удобно», «при необходимости», «и т.д.».
- `Uniform`: описаны состояния `EMPTY/LOADING/SUCCESS/ERROR` и сценарии ошибок `E201-E209`, `E301-E303`.
- `Scoped`: одна фича, объём менее 1500 слов, не более 3 модулей.
- Ссылка на Brief/Issue: указаны в разделе «Ссылки».
- Scope: есть разделы «Входит/Не входит».
- Инварианты: перечислены отдельным разделом.
- Ограничения реализации: перечислены отдельным разделом.

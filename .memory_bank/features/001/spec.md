# Выбор навыка и темы перед генерацией задачи

## Цель
Обеспечить обязательный шаг, на котором пользователь указывает навык и тему, чтобы сценарий генерации задачи мог быть запущен для 100% пользователей.

## Каноническое правило валидности
`skill` и `topic` считаются валидными, если после `strip_tags` и `trim` соответствуют regex `^(?!\W*$).{1,100}$`.

## Scope
Входит:
- Экран/форма с двумя обязательными полями: `Навык` и `Тема`.
- Валидация обязательности полей до запуска генерации задачи.
- Блокировка перехода к генерации задачи, пока поля пустые или не соответствуют каноническому правилу валидности.
- Передача `навыка` и `темы` в следующий шаг сценария генерации.

Не входит:
- Изменение логики генерации задачи, тестов или вердикта.
- История, аналитика, персонализация, рекомендации навыков/тем.
- Расширение на языки программирования, кроме Ruby.

## Требования

### Функциональные
- Пользователь видит форму с двумя полями ввода: `Навык` и `Тема`.
- Оба поля обязательны для продолжения.
- Кнопка перехода к генерации задачи недоступна, пока одно из полей пустое или не соответствует каноническому правилу валидности.
- При успешном заполнении пользователь может перейти к следующему шагу, где запускается генерация задачи.
- В следующий шаг передаются оба значения: `skill`, `topic`.
- При невалидном вводе пользователь получает сообщение об ошибке из таблицы `Сценарии` рядом с соответствующим полем.

### Нефункциональные
#### Производительность:
- Валидация полей выполняется не дольше 100 мс после изменения значения.
- Переход к следующему шагу после нажатия кнопки выполняется не дольше 1 секунды (p95).
#### Безопасность:
- Перед валидацией применяется `strip_tags` + `trim`; сохраняется очищенное значение.
- Если очищенное значение не проходит каноническое правило валидности, возвращается ошибка `Неверный формат. Поле должно содержать текст.`.
- Серверная валидация дублирует клиентскую и не допускает обход ограничений.

## Acceptance Criteria
- [ ] Пользователь не может перейти к генерации задачи, если `Навык` пустой.
- [ ] Пользователь не может перейти к генерации задачи, если `Тема` пустая.
- [ ] При заполненных `Навык` и `Тема`, соответствующих каноническому правилу валидности, переход к следующему шагу доступен.
- [ ] На следующем шаге доступны переданные значения `skill` и `topic`.
- [ ] При ошибке валидации пользователь видит сообщение из таблицы `Сценарии` у проблемного поля.

| E007 | Сервер вернул 5xx/timeout | `Не удалось перейти к генерации. Попробуйте ещё раз.` | Общий алерт формы | Да |

### Состояния

| State | Условие входа | UI | Кнопка `Сгенерировать` | Сетевое поведение | Условие выхода |
|------|----------------|----|------------------------|-------------------|----------------|
| EMPTY | Форма открыта впервые или после `reset`, оба поля пустые после `trim` | Пустые поля `Навык` и `Тема`, ошибок нет | Disabled | Запросы не отправляются | Пользователь изменил любое поле |
| ERROR | Возникла ошибка `E001-E007` | Для `E001-E006` сообщение под конкретным полем; для `E007` общий алерт формы; значения в полях сохраняются | `E001-E006`: disabled. `E007`: enabled, если поля валидны | Для `E001-E006` запрос не отправляется. Для `E007` разрешён повторный submit | Пользователь исправил данные (переход в `READY`) или нажал retry при `E007` (переход в `LOADING`) |
| READY | Оба поля валидны по каноническому правилу, ошибок нет | Поля заполнены, ошибок нет | Enabled | Запрос ещё не отправлен | Пользователь нажал `Сгенерировать` |
| LOADING | Пользователь нажал `Сгенерировать` в состоянии `READY` | На кнопке спиннер и текст `Генерация...`; поля и кнопка disabled | Disabled | Отправляется ровно 1 запрос перехода на следующий шаг с `skill`, `topic` | Ответ сервера: `2xx` -> `SUCCESS`, `4xx/5xx/timeout` -> `ERROR` |
| SUCCESS | Сервер вернул `2xx` и переход разрешён | Редирект на следующий шаг генерации задачи | Не применимо | Повторные запросы не отправляются | Терминальное состояние для текущего шага |

### Переходы между состояниями

| From | Событие | Условие | To |
|------|---------|---------|----|
| EMPTY | `input/change` | Оба поля валидны | READY |
| EMPTY | `input/change` | Любое поле невалидно | ERROR |
| ERROR | `input/change` | Оба поля валидны | READY |
| READY | `click submit` | Всегда | LOADING |
| LOADING | `response 2xx` | Всегда | SUCCESS |
| LOADING | `response 4xx/5xx/timeout` | Всегда | ERROR |
| ERROR | `click retry` | Только для `E007`, оба поля валидны | LOADING |


## Ограничения
### НЕ делать:
- Изменения в логике генерации самой задачи и тестов.
### Использовать:
- Service object для оркестрации перехода к шагу генерации (если бизнес-логика выходит за пределы контроллера).
- Существующие UI-компоненты формы и текущие механизмы валидации проекта.

## Инварианты:
- переход возможен только при skill/topic длиной 1..100 после нормализации;
- на следующий шаг всегда передаются оба поля;
- при любой ошибке значения формы не теряются;
- логика генерации задачи не изменяется этой фичей.
